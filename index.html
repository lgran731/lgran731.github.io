The Challenge

The program inherited an initial prototype from Army Research Laboratories that was completely non-viable: outdated model, zero documentation, no measurable accuracy, and no scalable architecture. The team needed real-time target detection and tracking across multiple camera streams to support vehicle operations.

Midway through development, executive stakeholders surfaced new requirements for infrared camera support that hadn't been communicated earlier, requiring scope reset with only 2 months until a major evaluation event.

My Role & Approach

I served as single-threaded owner for platform roadmap, architecture decisions, and delivery outcomes. I had final decision-making authority on Python → C++ migration, client-server architecture, and ML inference pipeline design.

I made the call to reset and rebuild the platform from the ground up, defining what "real-time" actually meant (30-40 FPS, <70ms latency) and what acceptance criteria we needed to hit.

I led cross-functional execution of 10-14 engineers and data scientists through Agile delivery with 2-week sprints, managing $1M budget and owning go/no-go readiness decisions.

Technical Solution

Model Selection

Led model trade studies across YOLO variants. Selected YOLO medium model to balance latency vs. accuracy constraints for real-time operations.

Architecture Migration

Drove Python → C++ migration for real-time constraints. Refactored monolithic design into distributed client-server model enabling multi-GPU workload distribution.

Performance Optimization

Implemented FP32 → FP16 optimization reducing GPU memory and bandwidth pressure. Redesigned tracking pipeline for multi-camera stitching and identity persistence.

Results & Impact

  • Scaled from 3 camera streams → 18 concurrent streams while maintaining real-time performance (40 FPS, <70ms latency)
  • Improved detection accuracy from ~30% baseline → >75% across 13 target classes in live video
  • Achieved ~2× throughput increase through architecture optimization and multi-GPU distribution
  • Reduced false detections that previously caused users to disable early prototypes
  • Delivered working IR camera support within 2-month compressed timeline after late requirement change
  • Platform now adopted as core dependency for all third-party model integrations and A/B testing
  • Established baseline architecture for future ruggedized ML model integrations with industry partners

The Challenge

Objective workload measurement required real-time analytics based on physiological data, but existing commercial solutions relied on device-generated metrics that were noisy, unverifiable, and unsuitable for real-time ML inference or system-level diagnostics.

The program needed transparent, debuggable analytics pipelines that could process raw biosignals in real-time, provide accurate workload assessments, and support iterative technology evaluation.

My Role & Approach

I led platform strategy and execution as primary technical and programmatic owner, making the critical decision to abandon derived device metrics in favor of raw signal processing for transparency and debuggability.

I prioritized deterministic execution over rapid prototyping by migrating critical pipelines from Python into compiled languages (C#) to meet latency and reliability constraints for production deployment.

Technical Solution

Signal Processing Pipeline

Built custom preprocessing pipeline: raw ECG ingestion, signal filtering, R-peak detection, HRV computation, baseline normalization. Improved signal quality 40% through custom preprocessing.

Python → C# Migration

Rewrote real-time pipeline from Python to C# to meet latency requirements. Used cross-validation between implementations to ensure model and signal fidelity.

ML Integration

Rebuilt ML inference layer using platform-compatible tooling (Microsoft ML) to replace research-only Python executables, enabling real-time workload visualization during live scenarios.

Results & Impact

  • Delivered production-ready workload analytics platform capable of real-time inference with live visualization
  • Platform supported 4 evaluation events, enabling objective technology assessment and data-informed design iteration
  • Established diagnostics and data-quality gating to mitigate sensor placement issues, signal degradation, and connection loss
  • Platform adoption enabled 4 active research partnerships through Cooperative Research and Development Agreements (CRADAs) with universities (U Michigan, Oakland, Clemson) and defense contractors (American Rheinmetall)
  • Created reusable architecture for integrating additional biosignals and ML models, establishing framework for objective workload evaluation
  • Platform now essential dependency for human factors and data analysis teams assessing technology impact on users
